📌 このドキュメントの方針

docs/021 第2部（トップ専門家2名 D・E のレビュー）で 「リスク高め」と判定された施策を除外し、 自社チャネル（PWA・プッシュ・メール・Telegram）を中心に据えた合法かつ凍結リスクの低い施策のみを、Phase 0 → 4 の順で並べた実行手順書です。

各施策は 021の施策番号 と紐付けてあります。上から順に着手すれば、測定基盤 → 生命線（自社チャネル）→ 外部流入 → SEO → バイラル の順で固まります。

⚠️ 本手順書から除外した施策（リスク高め）

  • 施策1：X 自動投稿（セール速報の cron 自動化） ― 再評価 3/10
    新規アカウント＋外部リンク＋アダルト＋bot投稿で凍結フラグが重なり、al.fanza.co.jp 直貼りは過去に大量凍結実績。アフィリンク自動投稿は最も凍結されやすいパターンのため除外。 ※ X を使う場合は「自社サイト(/sale)経由・センシティブ設定・頻度制御」を別途設計してから。本手順書では扱わない。
  • 施策8：5ch 等への体験談シード投稿 ― 再評価 2/10
    2023年10月施行のステマ規制（景表法）により、運営者が第三者を装って自社を宣伝する行為は違法。完全に除外し、合法な代替（施策10 ユーザー自発シェア）に投資する。

計測・埋もれ資産の起動

タイミング：今すぐ ／ 「測定なしに走らない。既存資産をタダで活かす」

▸ ゴール：どの施策が効いたか数字で判断できる土台を作り、実装ゼロで効く既存機能を起こす。

自前KPI計測（ファネル可視化）

追加12 優先 高 コスト 中

GA4 / Search Console はアダルトで規約上使えない・凍結されるケースがある。自前イベント計測を必須インフラとして先に作る。

  1. Supabase に events テーブルを追加（event_type / session_id / ref / item_id / created_at）。RLS はサービスロール書き込みのみ。
  2. 計測する6段階を定義：訪問 → Age通過 → PWA導入 → 通知許可 → アフィリンククリック → 再訪。
  3. 各イベント送信用の Route Handler app/api/track/route.ts を作成（既存 app/api/view/route.ts のパターンを流用）。
  4. 流入元計測：外部に出すURL全てに ?ref= / UTM を付与（X→?ref=x、メール→?ref=mail 等）。
  5. 集計用のシンプルなSQLビュー（Phase別の段階通過数）を用意し、週次で確認する。
events テーブルにイベントが記録され、流入元別に段階通過数が出せる。

既存バッジ機能をリテンション燃料に転用

追加14 優先 中 コスト 小

実装済みなのに使われていない lib/badges.ts を起こす。新規開発ゼロで再訪動機になる。

  1. lib/badge-engine.ts の付与ロジックが有効か確認（ポイントはコメントアウトのまま、バッジのみ）。
  2. マイページ / ホームにバッジ獲得状況と「次のバッジまであと◯」の進捗バーを表示。
  3. Phase 1 の通知と連携：「あと1日で7日連続ログインバッジ」のリマインド通知を notification_queue に積む。
狙い：STREAK_7 / COLLECTOR_50 などは再訪の強力な動機。通知（施策9）と組み合わせると効果が跳ねる。
バッジ進捗がUIに表示され、達成間近ユーザーへリマインド通知が飛ぶ。
1

自社チャネルの確立

タイミング：今週 ／ 「プラットフォームに殺されない資産を最優先で固める」

▸ ゴール：X や Google に依存せず、凍結されても生き残る「通知許可ユーザー」を増やす。ここが事業の生命線。

Web Push に「セール速報通知」訴求を追加

施策9 最優先 9/10 コスト 小〜中

現状の通知訴求は「推し女優の新作」のみ＝既存ファン向け。「毎日深夜に最大90%OFFのセールが通知で届く」軸を足すと、推し未定ユーザーの登録率が跳ねる。

  1. notification_subscriptions テーブルに notification_type カラムを追加（oshi / sale / both）。
  2. 通知登録フォームで「推し女優通知」「セール速報通知」を選べるUIにする。
  3. 登録フォームに 「現在 ◯人が登録中」 を Supabase の実数で表示（景表法上、必ず実データと連動）。
  4. 通知文面は熱量のある文体に：「{女優名}に待望の新作🎉 今だけ初日特典あり」。
法務注意：登録者数の表示は必ず実DB値を使う。水増しは景表法違反。
セール通知でも登録でき、登録者実数が表示され、type別に配信を出し分けられる。

PWA オンボーディング × 推し通知導線の改善

施策3 最優先 9/10 コスト 中

唐突なブラウザ通知プロンプトは拒否率80%超。事前の価値提示で許可率を上げる。プラットフォーム非依存の唯一の自社チャネルなので最優先。

  1. Age Check 通過直後の最初の1画面に、PWAインストール訴求＋推し女優登録フローを配置。
  2. 「推し登録 → 新作が出たらスマホに通知」を図解付きで説明してから許可ボタンを押させる。
  3. 通知許可を2ステップ化：自前モーダル（価値提示）→ ブラウザプロンプト。これで拒否率を下げる。
  4. 女優ページのヒーローセクション直下にも「推し登録 → 通知」CTAを設置（今は埋もれている）。
  5. 「PWAインストール方法」FAQページを用意（iPhone/Androidの手順）。「FANZA アプリ iPhone」検索流入も狙える。
Age通過後に価値提示モーダルが出て、2ステップで通知許可が取れる。通知許可率を計測（Phase 0）で追える。

「セール速報」を分散配信アーキテクチャ化

追加13 優先 高 コスト 中

施策9・施策3・Phase 2 の友好チャネルを束ねる上位戦略。1本の速報を複数チャネルへ同時配信し、どこかが凍結されても他で生き残る。

  1. 速報生成ロジックを1本化（既存の価格監視 Cloudflare Workers cron / fetchDailySaleItems 相当を流用）。
  2. 生成結果を ファンアウト：Web Push（施策9）→ メール → Telegram（Phase 2）へ同一データを配信。
  3. 配信先を増やしても生成は1箇所。配信アダプタだけ足す設計にする。
狙い：「FANZAセール速報の一次ソース」としてブランド化する。個別チャネルの凍結に一喜一憂しない。
1回の生成で複数チャネルに速報が配信される。1チャネル停止しても他は継続。

友好チャネルへの展開

タイミング：2週間以内 ／ 「敵対的だが大きい から 友好的で濃い へ振り直す」

▸ ゴール：アダルトを許容するチャネルに張り、凍結リスクなしで外部流入を作る。

アダルト許容チャネルへの集中投下

追加11 最優先 コスト 小〜中

一般チャネル（X/TikTok/Instagram/Google）がアダルトに敵対的なら、最初からアダルト許容チャネルに張るのが垂直グロースの定石。ペルソナ（オタク会社員）はここにも必ずいる。

  1. Telegram「セール速報チャンネル」を開設。アダルト配信が許容され、プッシュ性も高い。追加13の配信先として接続。
  2. 自社メールマガジンを用意（プラットフォーム非依存・一度取得すれば永続資産）。速報の配信先に追加。
  3. アダルトまとめ系・アンテナサイトへ相互リンク / RSS配信で送客（アダルトアフィリの王道）。
  4. FANZA/DMM エコシステム内の公式アフィリエイト導線・関連サービスからの送客を確認。
優先度：Telegram と メルマガ（自社資産）を先に。まとめサイトは相手探しに時間がかかるので並行で。
Telegramチャンネルとメルマガが稼働し、追加13の分散配信に組み込まれている。

SEO の仕込み

タイミング：Phase 1・2 と並行 ／ 「3ヶ月先の有機流入を今から仕込む」

▸ ゴール：SafeSearch の制約を理解し、ビッグワードは捨て「指名 × ロングテール」に集中する。

前提（D・E指摘）：アダルトは SafeSearch で一般クエリにほぼ表示されない。狙うのは指名性の高いクエリ（「女優名」「作品名」「FANZA セール」）のみ。汎用ビッグワードSEOへの色気は捨てる。

/sale デイリーSEO（タイトルに日付）

施策4 優先 高 7/10 コスト 小

「FANZA セール 今日」は毎日検索される。日付入りタイトルでフレッシュネスシグナルとCTRを上げる。低コスト高効果。

  1. /sale の generateMetadata で h1・description に今日の日付を動的挿入（例：「今日のFANZAセール・割引作品（2026年6月1日更新）」）。
  2. canonical は /sale に固定。日付はh1・descriptionのみ（old URLキャッシュ事故を防ぐ）。
  3. 「毎時更新」は ISR の事実なので description に明記してよい。
/sale が日付入りタイトルで表示され、canonical は固定されている。実装2時間以内。

女優ページ SEO 強化（指名クエリ）

施策2 優先 高 8/10 コスト 中

「[女優名] おすすめ 作品」は明確な購買意図クエリ＝SafeSearch下でも露出できる指名性。現状はグリッド中心でテキストが薄い。

  1. app/actress/[id]/page.tsx 本文に追加：h2「{名前}の人気作品ランキング」（review順TOP5）。
  2. h2「{名前}の関連女優」（共演頻度の集計済みデータを再利用）。
  3. generateMetadata の description：「{名前}のFANZA作品を全{N}本掲載。B{x} W{y} H{z}。人気ジャンルは{top3}。」
  4. 追加APIコールなし：既存の fetchActressList / fetchItemList のデータを再利用する。
女優ページに構造化テキストが入り、追加API呼び出しゼロで指名クエリ向けの本文が揃う。

ジャンル別 FAQ 充実（AI Overview 引用狙い）

施策6 優先 中 7/10 コスト 中

FAQPage 構造化データは実装済み。汎用文を、ジャンル名に合わせた具体的Q&Aに強化するとAI Overview引用・指名検索増に効く。

  1. 上位50ジャンルに絞り、ジャンル名に合わせた具体的Q&Aを手動で強化。
  2. FAQに「このサイトでできること（通知・スワイプ）」を含めて機能訴求も兼ねる。
  3. 「FANZAピックス」というブランド名が回答内に出るよう設計し、指名検索を増やす。
留意：AI Overviewはアダルト抑制傾向。ゼロクリックでもブランド認知に繋がる前提で割り切る。
上位50ジャンルのFAQが具体的Q&Aに差し替わっている。

比較・ガイドコンテンツ /guide

施策7 優先 中 6/10 コスト 中

「FANZA 使い方」「FANZA お得 使う方法」など情報収集フェーズを獲得。優先度は低めでよい。

  1. /guide を静的ページとして新設。「なぜこのサイトを使うか」のLP兼SEOコンテンツに。
  2. 比較軸：コンテンツ数 / 価格体系 / セール頻度・割引率 / スマホ体験（スワイプ機能が差別化）。
  3. 価格比較は静的記述。更新頻度の高い「今月のセール割引率」だけ DMM API で動的取得し鮮度を保つ。
/guide が公開され、動的部分は今月のセール割引率のみAPI取得。

バイラル（ユーザーを拡散エンジンに）

タイミング：継続 ／ 「ユーザー自身に広めてもらう」

▸ ゴール：合法かつ持続的な口コミ。運営の自作自演（施策8）は使わず、ユーザーの自発シェアに投資する。

お気に入りランキングのシェア機能

施策10 優先 中 7/10 コスト 小

ユーザーのお気に入りを「自分のFANZAランキング」として1クリックでシェア。ユーザー自身が拡散エンジンになる（合法な口コミ＝施策8の代替）。

  1. /favorites にシェアボタンを追加。app/actress/[id]/page.tsx の既存実装パターンを流用。
  2. twitter.com/intent/tweet に定型文＋URLを渡すだけ（TOP3＋★評価＋ハッシュタグ）。
  3. シェアURLに ?ref=share を付与し、Phase 0 の計測でリファラーを追える状態に。
狙い：「友人のお気に入り → 自分も試す」FOMO。直帰率の低さが鍵なので計測必須。
/favorites にシェアボタンがあり、?ref=share 経由の流入を計測できる。

スワイプ機能デモ動画（投稿先を限定）

施策5（改） 優先 中 5/10 コスト 小

「FANZA版Tinder」のスワイプUIは最大の WOW。ただし投稿先の選定が重要。X/TikTok/Instagramはアダルト禁止のため、そこへの投稿は避ける。

  1. スマホでスワイプ操作を15秒録画（録画15分・編集30分程度）。
  2. 冒頭「FANZA版Tinder作ってみた」→ スワイプ→LIKEエフェクト→お気に入り→ 最後にURL。
  3. 投稿先はアダルト許容チャネルに限定：Telegram / 自社サイト埋め込み / アダルトまとめ系。X・TikTok・Instagramには出さない。
  4. 最終フレームに「PWAで使える（インストール不要）」を添えて通知許可へ繋ぐ。
除外との境界：施策5自体は採用するが、021第2部の指摘どおり一般SNSへの投稿は凍結リスクのため行わない。
デモ動画がアダルト許容チャネルにのみ投稿され、URL経由の流入を計測できる。
